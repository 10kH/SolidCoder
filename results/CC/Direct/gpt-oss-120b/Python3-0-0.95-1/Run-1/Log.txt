**Solution Explanation**

For every book we are given a string `s` of length `m` (only capital letters `A … Z`).

For two different strings `a , b`

* look at the first position `p` where they differ (`1 ≤ p ≤ m`);
* if `p` is odd we want the **smaller** letter to come first (usual
  lexicographic order);
* if `p` is even we want the **larger** letter to come first
  (reverse order).

The required order is called **asc‑desc‑ending**.



--------------------------------------------------------------------

#### 1.   Transforming a string to a normal key  

If we could change every even position so that a *larger* original
letter becomes a *smaller* transformed letter,
then a **normal** lexicographic (ascending) comparison of the transformed
strings would give exactly the wanted order.

For a capital letter `c`

```
c = 'A' … 'Z'                (code 0 … 25)
rev(c) = 'Z' - (c - 'A')      # 'A' ↔ 'Z', 'B' ↔ 'Y', …
```

`rev` reverses the alphabet, therefore

```
original larger   →   rev smaller
original smaller  →   rev larger
```

Now we build a *key* for a string `s`

```
key[i] = s[i]                     if i is odd   (1‑based)
key[i] = rev(s[i])                if i is even  (1‑based)
```

`key` has the same length `m`.  
Sorting the keys with the ordinary ascending string order is exactly the
required asc‑desc‑ending order for the original strings.



--------------------------------------------------------------------

#### 2.   Algorithm
```
read n , m
prepare table rev[0..25]  # rev[x] = chr(ord('Z') - x)

pairs = []                     # (key , original_index)

for i = 1 … n
        read s
        build key:
                for each position j (0‑based)
                        if j is even   # odd position 1‑based
                                k = s[j]
                        else
                                k = rev[ ord(s[j]) - ord('A') ]
                key = ''.join(all k)
        pairs.append( (key , i) )          # i is 1‑based index

sort pairs by the first component (the key)

output the second components of the sorted pairs, separated by blanks
```

#### 3.   Correctness Proof  

We prove that the algorithm outputs the indices of the strings in the
required asc‑desc‑ending order.

---

##### Lemma 1  
For any two letters `x , y` (`A … Z`) and an even position `p`
(`p` 1‑based, i.e. `p` is even):

```
x < y   (alphabet order)   ⇔   rev(x) > rev(y)
```

**Proof.**  
`rev` maps `c` to `'Z' - (c-'A')`.  
If `x < y` then `c_x - c_y < 0`.  
Consequently `rev(x) - rev(y) = -(c_x - c_y) > 0`. ∎



##### Lemma 2  
Let `a , b` be two different original strings and `ka , kb` their
corresponding keys built by the algorithm.
Let `p` be the first position where `a` and `b` differ.
Then the relation between `ka[p]` and `kb[p]` is exactly the rule of the
asc‑desc‑ending order:

* if `p` is odd, `ka[p] < kb[p]`  ⇔  `a[p]` is alphabetically smaller;
* if `p` is even, `ka[p] < kb[p]`  ⇔  `a[p]` is alphabetically larger.

**Proof.**  
If `p` is odd, the algorithm copies the original character, therefore
`ka[p] = a[p] , kb[p] = b[p]` and the statement holds trivially.

If `p` is even, the algorithm stores `rev(a[p]) , rev(b[p])`.
By Lemma&nbsp;1, `a[p]` alphabetically larger ⇔ `rev(a[p])` alphabetically
smaller, i.e. `ka[p] < kb[p]`. ∎



##### Lemma 3  
For any two distinct strings `a , b` :

```
a precedes b in asc‑desc‑ending order   ⇔   key(a) < key(b)   (usual string order)
```

**Proof.**  
Let `p` be the first differing position.
By definition of asc‑desc‑ending order, the rule applied at `p`
determines which string comes first.
By Lemma&nbsp;2 the same rule determines the order of the characters
`key(a)[p]` and `key(b)[p]`.  
All earlier positions are equal in both original strings and in the keys,
hence the whole keys are ordered exactly as the original strings. ∎



##### Lemma 4  
After sorting the list `pairs` by the key component,
the sequence of stored indices is the sequence of original strings sorted
in asc‑desc‑ending order.

**Proof.**  
Sorting by the key component yields a list where for any consecutive
elements `(key_i , idx_i) , (key_{i+1} , idx_{i+1})` we have
`key_i ≤ key_{i+1}`.
By Lemma&nbsp;3 this is equivalent to the original strings being ordered
according to asc‑desc‑ending. ∎



##### Theorem  
The algorithm outputs exactly the indices of the given strings after they
are sorted asc‑desc‑endingly.

**Proof.**  
The algorithm builds for each input string its key (definition of the
key).  
It stores the pair `(key , original_index)` for every string.
By Lemma&nbsp;4 sorting these pairs by the key produces the indices in the
desired order.
Finally the algorithm prints those indices, therefore the output is
correct. ∎

#### 4.   Complexity Analysis  

*Let `N = n` and `M = m`.*

* Building all keys: each character is processed once  
  `O(N·M)` time, `O(N·M)` total characters stored (the keys).  
  Because `N·M ≤ 10⁶`, this fits easily into memory.

* Sorting `N` items with keys of length `M`:
  Python’s Timsort uses `O(N log N)` comparisons, each comparison of two
  keys costs at most `O(M)` but the keys are already stored, so the
  overall time is `O(N log N)` (the factor `M` is hidden inside the
  string comparison implementation).  
  With the given limits this is well below one second.

* Memory consumption:  
  - the list of `N` pairs, each pair stores a reference to a key string
    (`M` characters) and an integer → `O(N·M)` characters + overhead.  
  - no other large structures.  
  ≤ a few megabytes for the maximal input (`10⁶` characters).

**Solution Explanation**

For one turn Casimir can erase  

* one `A` together with one `B`, **or**  
* one `B` together with one `C`.

The two letters do **not** have to be adjacent, therefore the only thing that matters is
*how many* letters of each kind are present, not where they are.

--------------------------------------------------------------------

#### 1.   Observations  

* Every turn removes **exactly one `B`** and **exactly one non‑`B`** (`A` or `C`).
* After all turns the whole string is empty, therefore the total number of removed
  `B`’s equals the total number of removed non‑`B`’s.

Let  

```
cntA = number of 'A' in s
cntB = number of 'B' in s
cntC = number of 'C' in s
```

All letters are erased  **iff**

```
cntB = cntA + cntC                (1)
```

Because then we can pair each `A` with a distinct `B` and each `C` with a distinct
remaining `B`. The order of the pairings is irrelevant – we may always pick any
still existing `B` together with any still existing `A` or `C`.

Condition (1) is also necessary: if the string becomes empty, the number of erased
`B`’s (which is `cntB`) must equal the number of erased `A`+`C` (which is `cntA+cntC`).

A side consequence is that the length of the string must be even, but this is already
guaranteed by (1) because  

```
|s| = cntA + cntB + cntC = (cntA+cntC) + cntB = 2·cntB .
```

So the whole problem reduces to a single count comparison.

#### 3.   Correctness Proof  

We prove that the algorithm outputs **YES** exactly for the strings that can be
completely erased.

---

##### Lemma 1  
If a string can be completely erased, then `cntB = cntA + cntC`.

**Proof.**  
Each turn removes one `B` and one non‑`B`. After all turns the total number of removed
`B`’s equals the total number of removed `A`+`C`. Since finally no letters remain,
the numbers of removed letters are exactly the original counts, i.e. `cntB = cntA+cntC`. ∎



##### Lemma 2  
If `cntB = cntA + cntC` for a string `s`, then `s` can be completely erased.

**Proof.**  
Because `cntB` equals the total number of `A` and `C`, we can perform the following
sequence of turns:

* Pair each of the `cntA` letters `A` with a distinct `B` (possible because there are at
  least `cntA` unused `B`’s).
* Pair each of the `cntC` letters `C` with a distinct remaining `B`.

All pairings are allowed by the rules, and after `cntA+cntC = cntB` turns every letter
has been removed. ∎



##### Theorem  
For every test case the algorithm prints **YES** iff the given string can be fully
erased.

**Proof.**  

*If the algorithm prints YES.*  
Then `cntB = cntA + cntC`. By Lemma&nbsp;2 the string can be erased.

*If the string can be erased.*  
By Lemma&nbsp;1 we have `cntB = cntA + cntC`, therefore the algorithm’s condition
holds and it prints YES.

Thus the output of the algorithm coincides exactly with the true answer. ∎

**Solution Explanation**

For every cell we know whether it already is an obsidian block (`1`) or an empty
block (`0`).  
In one operation we may change the type of a cell.

--------------------------------------------------------------------

#### 1.   Portal definition  

A sub‑rectangle `M` of size `a × b` (`a ≥ 5 , b ≥ 4`) is a *portal* iff  

* the whole **border** (except the four corners) consists of obsidian blocks,
* every **inner** cell (rows `2 … a‑1`, columns `2 … b‑1`) is empty,
* the four corner cells may be anything.

For a fixed rectangle we can count how many cells are wrong

```
border cells that are 0   → have to be flipped to 1
inner  cells that are 1   → have to be flipped to 0
```

The sum of those two numbers is the number of operations needed for this
rectangle.



--------------------------------------------------------------------

#### 2.   Prefix sums – O(1) queries  

`grid[i][j]` – `0` or `1` (0‑based indices).  
`pref[i+1][j+1]` – number of `1`s in the sub‑matrix
`[0 … i] × [0 … j]`.

With the usual 2‑D prefix sum we can obtain the number of `1`s in any
axis‑parallel rectangle in **O(1)** :

```
ones(r1, c1, r2, c2)   # rows [r1, r2) , cols [c1, c2)
    = pref[r2][c2] - pref[r1][c2] - pref[r2][c1] + pref[r1][c1]
```

--------------------------------------------------------------------

#### 3.   Fix the top and bottom rows  

Let the top row be `top` and the bottom row be `bot`
(`bot - top + 1 = a ≥ 5`).  
All rows between them are *inner rows*.

For every column `c` we pre‑compute three values

```
innerOnes[c]   = number of 1's in rows (top+1 … bot-1) of column c
borderZero[c] = (a-2) - innerOnes[c]          # zeros that must become 1
topBotZero[c] = (grid[top][c] == 0) + (grid[bot][c] == 0)
                # zeros on the top and bottom edge of the rectangle
```

*If a column becomes the left or right border* we have to pay `borderZero[c]`
(because the whole vertical side must be `1`).  
*If a column is an inner column* we have to pay

```
topBotZero[c]          # top & bottom edge cells must be 1
+ innerOnes[c]         # inner cells must be 0
```

--------------------------------------------------------------------

#### 4.   From columns to a sliding window  

Consider a window of columns `[L … R]` (`R-L+1 = b ≥ 4`).

```
cost(L,R) = borderZero[L] + borderZero[R]                # left & right side
          + Σ_{c = L+1}^{R-1} ( topBotZero[c] + innerOnes[c] )
```

Define prefix sums over the columns

```
prefTB[x] = Σ_{c < x} topBotZero[c]
prefIn[x] = Σ_{c < x} innerOnes[c]
```

Then

```
Σ_{c = L+1}^{R-1} topBotZero[c] = prefTB[R] - prefTB[L+1]
Σ_{c = L+1}^{R-1} innerOnes[c] = prefIn[R] - prefIn[L+1]
```

so

```
cost(L,R) = borderZero[R] + prefTB[R] + prefIn[R]
            + ( borderZero[L] - prefTB[L+1] - prefIn[L+1] )
```

For a fixed right border `R` the part in parentheses depends **only** on `L`.
Therefore while we sweep `R` from left to right we keep the minimum value of

```
value(L) = borderZero[L] - prefTB[L+1] - prefIn[L+1]
```

over all admissible `L ≤ R-3` (because width must be at least `4`).

The algorithm for one pair `(top,bot)` :

```
best = +∞
for R = 3 … m-1:
        L = R-3                     # the newest L that becomes allowed
        best = min( best,
                    borderZero[L] - prefTB[L+1] - prefIn[L+1] )
        cur = borderZero[R] + prefTB[R] + prefIn[R] + best
        answer = min(answer, cur)
```

All operations inside the loop are O(1).  
The loop itself is `O(m)`.

--------------------------------------------------------------------

#### 5.   Whole algorithm  

```
answer = +∞
for top = 0 … n-5:
    for bot = top+4 … n-1:                # height a = bot-top+1 ≥ 5
        compute the three column arrays (innerOnes, borderZero, topBotZero)
        build their column prefix sums (prefTB, prefIn)
        run the O(m) sliding window described above
        update answer
print answer
```

Number of row pairs: `O(n²)` ( ≤ 80 000 for `n = 400`).  
For each pair we spend `O(m)` ( ≤ 400 ) time.  
Overall complexity per test case  

```
Time   :  O(n² · m)   ≤ 3.2·10⁷ elementary operations
Memory :  O(n·m) for the 2‑D prefix sum ( ≤ 1.6·10⁵ integers )
```

Both are easily fast enough for the given limits.

#### 6.   Correctness Proof  

We prove that the algorithm outputs the minimum possible number of operations.

---

##### Lemma 1  
For a fixed rectangle `R` the number of required operations equals  

```
(# of border cells that are 0) + (# of inner cells that are 1)
```

**Proof.**  
All border cells (except corners) must be `1`.  
If a border cell already is `1` nothing has to be done, otherwise it must be
flipped – exactly one operation per such cell.  
All inner cells must be `0`.  
If an inner cell already is `0` nothing is needed, otherwise it must be
flipped – again one operation per such cell.  
Corners are unrestricted, they never contribute. ∎



##### Lemma 2  
For a fixed pair of rows `(top, bot)` and any column `c`

* `borderZero[c]` is exactly the number of operations needed on column `c`
  **if** `c` becomes a left or right side of the portal,
* `topBotZero[c] + innerOnes[c]` is exactly the number of operations needed on
  column `c` **if** `c` is an inner column of the portal.

**Proof.**  
The vertical side consists of the cells in rows `top+1 … bot-1`.  
Among them `innerOnes[c]` are already `1`, the remaining `a-2-innerOnes[c]`
are `0` and have to be flipped – this is `borderZero[c]`.  

If `c` is an inner column, the two cells `(top,c)` and `(bot,c)` belong to the
top / bottom border, therefore each zero among them costs one operation –
that is `topBotZero[c]`.  
All cells between `top` and `bot` belong to the interior, each `1` among them
must be turned to `0` – exactly `innerOnes[c]` operations. ∎



##### Lemma 3  
For a fixed pair of rows `(top, bot)` and any column window `[L … R]`
(`R-L+1 ≥ 4`) the formula  

```
cost(L,R) = borderZero[R] + prefTB[R] + prefIn[R]
            + ( borderZero[L] - prefTB[L+1] - prefIn[L+1] )
```

equals the number of operations required to turn that sub‑rectangle into a
portal.

**Proof.**  
By Lemma&nbsp;2 the total cost is

```
borderZero[L] + borderZero[R]                     (left & right side)
+ Σ_{c = L+1}^{R-1} ( topBotZero[c] + innerOnes[c] )
```

Replace the sum with prefix sums:

```
Σ_{c = L+1}^{R-1} topBotZero[c] = prefTB[R] - prefTB[L+1]
Σ_{c = L+1}^{R-1} innerOnes[c] = prefIn[R] - prefIn[L+1]
```

Insert them, regroup the terms and obtain exactly the claimed expression. ∎



##### Lemma 4  
During the sweep over `R` the variable `best` always holds  

```
best = min_{L ≤ R-3} ( borderZero[L] - prefTB[L+1] - prefIn[L+1] )
```

**Proof.**  
Initially `best = +∞`.  
When `R` increases by one, the only new admissible left border is `L = R-3`.
The algorithm updates `best` with the value for this `L`.  
All previously admissible `L` remain admissible, and `best` already stores the
minimum among them. Hence after the update `best` is the minimum over the whole
set `{ L | L ≤ R-3 }`. ∎



##### Lemma 5  
For a fixed pair of rows `(top, bot)` the algorithm finds the minimum
`cost(L,R)` over all column windows of width at least `4`.

**Proof.**  
For each `R` (right border) the algorithm computes

```
cur = borderZero[R] + prefTB[R] + prefIn[R] + best
```

By Lemma&nbsp;4, `best` equals the minimum possible value of the second
summand in Lemma&nbsp;3 over all admissible `L`.  
Therefore `cur` equals the minimum `cost(L,R)` for this fixed `R`.  
Taking the minimum of `cur` over all `R` yields the global minimum over all
pairs `(L,R)` with `R-L+1 ≥ 4`. ∎



##### Lemma 6  
For a fixed pair of rows `(top, bot)` the algorithm returns the minimum number
of operations needed to obtain a portal whose top side is `top` and bottom side
is `bot`.

**Proof.**  
All portals with these two rows are exactly the column windows of width at
least `4`.  
By Lemma&nbsp;5 the algorithm finds the cheapest among them, i.e. the minimum
possible number of operations for this pair of rows. ∎



##### Lemma 7  
The overall answer returned by the algorithm is the minimum number of
operations needed to create **any** portal in the whole grid.

**Proof.**  
Every portal has a unique top row `top` and bottom row `bot` with
`bot - top + 1 ≥ 5`.  
The algorithm enumerates **all** such pairs of rows and, by Lemma&nbsp;6,
computes the cheapest portal for each pair.  
Taking the minimum over all pairs therefore yields the cheapest portal among
all possible portals. ∎



##### Theorem  
For each test case the program outputs the minimum possible number of
operations required to make at least one sub‑rectangle a portal.

**Proof.**  
Directly from Lemma&nbsp;7. ∎

```python
import sys

INF = 10 ** 9

def solve() -> None:
    it = iter(sys.stdin.read().strip().split())
    t = int(next(it))
    out_lines = []
    for _ in range(t):
        n = int(next(it))
        m = int(next(it))
        grid = [list(map(int, list(next(it).strip()))) for _ in range(n)]

        # 2‑D prefix sum of ones
        pref = [[0] * (m + 1) for _ in range(n + 1)]
        for i in range(n):
            row_sum = 0
            for j in range(m):
                row_sum += grid[i][j]
                pref[i + 1][j + 1] = pref[i][j + 1] + row_sum

        def ones(r1, c1, r2, c2):
            """number of 1's in rows [r1, r2) , cols [c1, c2)"""
            return pref[r2][c2] - pref[r1][c2] - pref[r2][c1] + pref[r1][c1]

        answer = INF

        # enumerate top and bottom rows
        for top in range(n - 4):          # need at least 5 rows
            for bot in range(top + 4, n):
                inner_len = bot - top - 1          # a-2
                # column wise data
                inner_one = [0] * m
                border_zero = [0] * m
                topbot_zero = [0] * m
                for c in range(m):
                    inner_one[c] = ones(top + 1, c, bot, c + 1)
                    border_zero[c] = inner_len - inner_one[c]
                    topbot_zero[c] = (grid[top][c] == 0) + (grid[bot][c] == 0)

                # prefix sums over columns
                pref_tb = [0] * (m + 1)
                pref_in = [0] * (m + 1)
                for c in range(m):
                    pref_tb[c + 1] = pref_tb[c] + topbot_zero[c]
                    pref_in[c + 1] = pref_in[c] + inner_one[c]

                best = INF
                # slide right border R, width >= 4  -> R starts from 3 (0‑based)
                for R in range(3, m):
                    Lcand = R - 3
                    expr = border_zero[Lcand] - pref_tb[Lcand + 1] - pref_in[Lcand + 1]
                    if expr < best:
                        best = expr
                    cur = border_zero[R] + pref_tb[R] + pref_in[R] + best
                    if cur < answer:
                        answer = cur

        out_lines.append(str(answer))
    sys.stdout.write("\n".join(out_lines))

if __name__ == "__main__":
    solve()
```

* an **ascent** is an index `i (1 ≤ i < m)` with `p[i] < p[i+1]`
* `A(m, k)` – the *Eulerian number* – counts permutations of length `m`
  that have exactly `k` ascents.

The required answer for a given `n` is  

```
ans(n) = Σ A(2n, k)   (k = n … 2n‑1)
```

--------------------------------------------------------------------

#### 1.  Symmetry of Eulerian numbers  

Eulerian numbers satisfy  

```
A(m, k) = A(m, m‑1‑k)          (0 ≤ k ≤ m‑1)
```

Proof sketch: reverse a permutation.
If a permutation has `k` ascents, the reversed permutation has
`(m‑1‑k)` ascents, giving a bijection.

--------------------------------------------------------------------

#### 2.  Half of all permutations  

For `m = 2n` the set `{0,…,m‑1}` is split into two equal parts

```
{0,…,n‑1}   and   {n,…,m‑1}
```

Because of the symmetry `A(m,k) = A(m,m‑1‑k)`,
every permutation counted in the first part is paired with a
different permutation counted in the second part.
No permutation is left unpaired ( `m‑1` is odd, therefore there is
no “middle” value `k = (m‑1)/2`).

Consequently

```
Σ_{k=n}^{m‑1} A(m,k) = ½ · Σ_{k=0}^{m‑1} A(m,k) = ½ · m!
```

(the sum of all Eulerian numbers for fixed `m` equals `m!`).

Therefore

```
ans(n) = (2n)! / 2   (mod MOD)
```

--------------------------------------------------------------------

#### 3.  Computing the answer  

* Pre‑compute factorials `fac[i] = i! (mod MOD)` for `i = 0 … 2·Nmax`,
  where `Nmax = max n` among all test cases (`≤ 10⁵`).
* The modular inverse of `2` is `inv2 = (MOD+1)//2` because `MOD` is an odd
  prime (`10⁹+7`).
* For each test case output  

```
ans = fac[2·n] * inv2  mod MOD
```

--------------------------------------------------------------------

#### 4.  Correctness Proof  

We prove that the algorithm outputs the required number of permutations
for every test case.

---

**Lemma 1**  
For any `m ≥ 1` and `0 ≤ k ≤ m‑1` we have `A(m,k) = A(m,m‑1‑k)`.

*Proof.* Reversing a permutation of length `m` turns each ascent into a
descent and each descent into an ascent.
If the original permutation has `k` ascents, the reversed one has
`(m‑1‑k)` ascents.
The reversal operation is a bijection, therefore the two sets have the
same size. ∎



**Lemma 2**  
For even `m = 2n`  

```
Σ_{k=n}^{m‑1} A(m,k) = (m)! / 2 .
```

*Proof.* By Lemma&nbsp;1 the terms with indices `k` and `m‑1‑k`
are equal.
The index set `{0,…,m‑1}` is partitioned into the two disjoint subsets
`L = {0,…,n‑1}` and `R = {n,…,m‑1}`.
The map `k ↦ m‑1‑k` is a bijection from `L` onto `R`.
Hence the total sum of all Eulerian numbers splits into two equal halves:

```
Σ_{k=0}^{m‑1} A(m,k) = Σ_{k∈L} A + Σ_{k∈R} A = 2· Σ_{k∈R} A .
```

Since the left‑hand side equals `m!`, we obtain the claim. ∎



**Lemma 3**  
For any `n ≥ 1` the number of permutations of length `2n` that have at
least `n` ascents equals `(2n)! / 2`.

*Proof.* Directly from Lemma&nbsp;2 with `m = 2n`. ∎



**Theorem**  
For each test case the algorithm outputs the number of permutations of
length `2n` whose number of ascents is **not less than** `n`, modulo
`10⁹+7`.

*Proof.*  
The algorithm computes `fac[2n] = (2n)! (mod MOD)` and multiplies it by
`inv2`, the modular inverse of `2`. Because `MOD` is prime,
`inv2` satisfies `2·inv2 ≡ 1 (mod MOD)`, therefore

```
output ≡ (2n)! · inv2 ≡ (2n)! / 2   (mod MOD).
```

By Lemma&nbsp;3 this value equals the exact count of required permutations.
Thus the output is correct. ∎



--------------------------------------------------------------------

#### 5.  Complexity Analysis  

*Precomputation* (once):

```
O(2·Nmax)   time
O(2·Nmax)   memory
```

*Each test case*: `O(1)` time, `O(1)` extra memory.

With `Σ n ≤ 10⁵` the limits are easily satisfied.

```python
import sys

MOD = 1_000_000_007
INV2 = (MOD + 1) // 2          # modular inverse of 2 modulo MOD

def solve() -> None:
    data = sys.stdin.read().strip().split()
    if not data:
        return
    t = int(data[0])
    ns = list(map(int, data[1:1 + t]))
    max_n = max(ns)

    # pre‑compute factorials up to 2*max_n
    limit = 2 * max_n
    fac = [1] * (limit + 1)
    for i in range(2, limit + 1):
        fac[i] = fac[i - 1] * i % MOD

    out_lines = []
    for n in ns:
        ans = fac[2 * n] * INV2 % MOD
        out_lines.append(str(ans))

    sys.stdout.write("\n".join(out_lines))

if __name__ == "__main__":
    solve()
```

**Solution Explanation**

For one test case we are given  

* `a` songs of length `1`
* `b` songs of length `2`
* `c` songs of length `3`

All songs have to be divided into two concerts.
If the total length of the first concert is `x`, the second one has length `S‑x`
(`S = a·1 + b·2 + c·3`).  
The difference of the concerts is  

```
| (S‑x) – x | = | S – 2·x |
```

So we have to choose a reachable value `x` ( `0 ≤ x ≤ S` ) that minimises
`|S‑2·x|`.

--------------------------------------------------------------------

#### 1.   When is a value `x` reachable ?

`x` is reachable iff we can write it as  

```
x = a1 + 2·b1 + 3·c1
```

with  

```
0 ≤ a1 ≤ a ,   0 ≤ b1 ≤ b ,   0 ≤ c1 ≤ c .
```

Only three different song lengths exist, therefore we can test the
reachability of a concrete `x` by trying a *few* possibilities for `c1`
and `b1`.  
Why only a few?  
If we take as many `3`‑minute songs as possible (`cmax = min(c, x//3)`)
the remaining amount `r = x – 3·cmax` has to be built from `1` and `2`.
If that fails we can decrease `cmax` by `1`.  
Decreasing `cmax` adds `3` to the remainder – this can be compensated by
changing the number of `2`‑minute songs by at most `2` (because `2·2 = 4`
covers the whole interval of three consecutive numbers).  
Consequently it is sufficient to try

```
c1 = cmax , cmax-1 , cmax-2
b1 = bmax , bmax-1 , bmax-2
```

where `bmax = min(b, (x-3·c1)//2)`.  
If for one of those pairs the remaining amount fits into the available
`1`‑minute songs (`remaining ≤ a`) the value `x` is reachable.

The test needs at most `3·3 = 9` very cheap operations.

--------------------------------------------------------------------

#### 2.   Searching the best `x`

The optimal `x` is close to `S/2`.  
The maximal possible difference is the length of a single song (`3`),
because we could put *all* songs into one concert and the other concert
would be empty.  
Therefore it is enough to look at the candidates

```
S//2 - d   and   S//2 + d      for d = 0,1,2,3
```

(their number is at most eight).  
For each candidate we call the reachability test described above.
The first candidate that is reachable gives the minimal possible
difference `|S‑2·x|`.

The whole algorithm for one test case works in **O(1)** time.

--------------------------------------------------------------------

#### 3.   Correctness Proof  

We prove that the algorithm prints the minimal possible difference.

---

##### Lemma 1  
For a fixed target value `x` the procedure `can(x)` (the 9‑iteration test)
returns *True* iff there exist integers `a1,b1,c1` respecting the limits
`0≤a1≤a , 0≤b1≤b , 0≤c1≤c` and `a1+2·b1+3·c1 = x`.

**Proof.**  
Let `cmax = min(c, x//3)`. Any feasible solution uses some `c1 ≤ cmax`.
If a feasible solution uses `c1 = cmax - k` with `k ≥ 3`,
replace it by `c1' = cmax - (k-3)`. The remainder grows by `3·3 = 9`,
which can be compensated by increasing `b1` by at most `5`
(`5·2 = 10 > 9`). Repeating this argument we can always transform a
solution into another one with `c1 ∈ {cmax, cmax-1, cmax-2}` while staying
inside the bounds for `b1` and `a1`.  
The same argument applied to the `2`‑minute songs shows that we may also
assume `b1 ∈ {bmax, bmax-1, bmax-2}` where `bmax = min(b, (x-3·c1)//2)`.  
Thus a feasible solution exists **iff** one of the at most nine examined
pairs `(c1,b1)` leaves a remainder not larger than `a`. ∎



##### Lemma 2  
If a value `x` is reachable, then every integer `y` with  
`|y‑x| ≤ 3` and `0 ≤ y ≤ S` is also reachable **or** the algorithm will
find a reachable value among the eight candidates
`⌊S/2⌋±d (d = 0…3)`.

**Proof.**  
Take any reachable `x`. Decreasing `x` by `1` can be realised by moving a
`1`‑minute song from the first to the second concert (possible only if
`a1>0`). If no `1`‑minute song is left, we can replace one `3`‑minute song
by three `1`‑minute songs – this changes the sum by exactly `3`.  
Analogously we can increase the sum by `1` using a `1`‑minute song or by
replacing three `1`‑minute songs with one `3`‑minute song.  
Consequently any integer within distance `3` from a reachable value is
also reachable, unless we run out of songs of the needed type.
If the optimal value `x*` (the one that minimises `|S‑2·x|`) is farther
than `3` from `⌊S/2⌋`, then `|S‑2·x*|` is at least `4`.  
But putting **all** songs into one concert gives a difference of at most
`3` (the longest single song). Hence the optimal difference can never be
larger than `3`, i.e. `x*` is at distance ≤ 3 from `⌊S/2⌋`. ∎



##### Lemma 3  
Among the eight numbers  
`candidates = {⌊S/2⌋‑d , ⌊S/2⌋+d | d = 0,1,2,3}` the algorithm finds the
reachable one with the smallest `|S‑2·x|`.

**Proof.**  
The algorithm checks the candidates in increasing order of `d`.  
The first reachable candidate has the smallest distance `d` to `⌊S/2⌋`,
hence its absolute difference `|S‑2·x|` is minimal among all candidates.
By Lemma&nbsp;2 the optimal value `x*` belongs to this set, therefore the
first reachable candidate is optimal. ∎



##### Lemma 4  
The difference printed by the algorithm equals the minimal possible
difference of the two concerts.

**Proof.**  
Let `x_alg` be the candidate selected by the algorithm.
By Lemma&nbsp;1 `x_alg` is reachable, therefore a partition with
difference `|S‑2·x_alg|` exists.  
By Lemma&nbsp;3 no other reachable value yields a smaller difference.
Consequently `|S‑2·x_alg|` is the global optimum. ∎



##### Theorem  
For every test case the program outputs the minimal possible difference
(in minutes) between the total lengths of the two concerts.

**Proof.**  
Directly from Lemma&nbsp;4. ∎

For each test case  

* at most **8** candidates are examined,
* for each candidate at most **9** simple arithmetic checks are performed.

Hence  

```
Time   :  O(1) per test case   ( ≤ 72 elementary operations )
Memory :  O(1)
```

Both limits easily satisfy the required constraints (`t ≤ 1000`).

```python
import sys

def solve() -> None:
    it = iter(sys.stdin.read().strip().split())
    t = int(next(it))
    out_lines = []

    for _ in range(t):
        a = int(next(it))
        b = int(next(it))
        c = int(next(it))

        total = a + 2 * b + 3 * c
        half = total // 2

        # ---------- reachability test ----------
        def can(x: int) -> bool:
            # try up to 3 different numbers of 3‑minute songs
            c_max = min(c, x // 3)
            for dc in range(3):
                c1 = c_max - dc
                if c1 < 0:
                    break
                rem1 = x - 3 * c1
                # now we need rem1 = a1 + 2*b1
                b_max = min(b, rem1 // 2)
                for db in range(3):
                    b1 = b_max - db
                    if b1 < 0:
                        break
                    rem2 = rem1 - 2 * b1
                    if rem2 <= a:          # a1 = rem2, must be non‑negative
                        return True
            return False
        # -----------------------------------------

        best = total                     # large initial value
        # examine candidates half-d, half+d  (d = 0..3)
        for d in range(4):
            # first the lower candidate, then the upper one
            for cand in (half - d, half + d):
                if cand < 0 or cand > total:
                    continue
                if can(cand):
                    diff = abs(total - 2 * cand)
                    if diff < best:
                        best = diff
            if best == 0:                # cannot be improved
                break

        out_lines.append(str(best))

    sys.stdout.write("\n".join(out_lines))

if __name__ == "__main__":
    solve()
```

**Solution Explanation**

For every test case we are given `n ( ≥ 3 )` distinct positive integers  
`a1 , a2 , … , an`.  
We have to choose a subset with the **maximum possible size** whose sum is a
*composite* number (i.e. not prime and larger than 1).

--------------------------------------------------------------------

#### 1.  Observations  

* Let  

```
S = a1 + a2 + … + an          (sum of the whole array)
```

*If `S` is already composite* – we can take the whole array, its size `n`
is obviously maximal.

*If `S` is prime* – we have to discard some elements.
We would like to discard as few as possible, because the remaining set
must be as large as possible.

* `S` is a prime ≥ 3, therefore `S` is **odd**.  
  (The only even prime is `2`, but `S ≥ 1+2+3 = 6`.)

* Removing an **odd** element from an odd sum makes the new sum even:

```
S (odd) – odd = even
```

* Every even number larger than `2` is composite
(because it is divisible by `2`).  
  After removing one odd element the new sum is at least `4`
(because the original sum is at least `7`), hence it is composite.

*If the whole sum is prime, the array must contain at least one odd element*,
otherwise the sum of only even numbers would be even and could not be an
odd prime.

Consequences  

* If `S` is prime we can obtain a composite sum by **removing exactly one
odd element**.  
  The remaining set has size `n‑1`, which is the largest possible size
  for a composite‑sum subset (we cannot keep all `n` elements because their
  sum is prime).

Thus the whole problem reduces to:

```
if total sum is composite → answer = all indices
else                         → answer = all indices except any odd element
```

--------------------------------------------------------------------

#### 2.  Algorithm
For each test case  

```
read n and the array a[1…n]
S = sum(a)

if S is not prime:
        output n
        output indices 1 … n
else:
        find an index i with a[i] odd
        output n-1
        output all indices except i
```

The only non‑trivial operation is primality testing of `S`.
`S ≤ 100 * 200 = 20000`, therefore a simple trial division up to `√S`
( ≤ 142 ) is more than fast enough.

--------------------------------------------------------------------

#### 3.  Correctness Proof  

We prove that the algorithm always outputs a largest‑size subset whose sum
is composite.

---

##### Lemma 1  
If the total sum `S` of the whole array is composite, the whole array is a
valid subset of maximum size `n`.

**Proof.**  
The whole array itself is a subset, its sum equals `S` which is composite.
No subset can be larger than `n`. ∎



##### Lemma 2  
If `S` is prime, the array contains at least one odd element.

**Proof.**  
Assume the contrary: all `a[i]` are even.  
The sum of even numbers is even, therefore `S` would be even.
The only even prime is `2`, but `S ≥ 1+2+3 = 6`. Contradiction. ∎



##### Lemma 3  
Let `S` be prime and `a[k]` be any odd element.
Then `S' = S – a[k]` is composite.

**Proof.**  
`S` is odd, `a[k]` is odd ⇒ `S'` is even.  
Since `S ≥ 7` (the smallest odd prime ≥ 3 and the sum of at least three
positive distinct numbers), we have `S' ≥ S – max(a[i]) ≥ 7 – 200`?  
More directly, the smallest possible `S` that is an odd prime with at least
three distinct positive integers is `7` (e.g. `1+2+4`).  
Removing an odd element (at least `1`) leaves `S' ≥ 6`.  
Every even integer `≥ 4` is divisible by `2`, hence composite.
Thus `S'` is composite. ∎



##### Lemma 4  
If `S` is prime, any subset of size `n‑1` obtained by deleting a single odd
element has a composite sum.

**Proof.** Directly from Lemma&nbsp;3. ∎



##### Lemma 5  
If `S` is prime, no subset of size `n` has a composite sum.

**Proof.** The only subset of size `n` is the whole array, whose sum is `S`,
prime by assumption. ∎



##### Theorem  
For every test case the algorithm outputs a subset of maximum possible
size whose sum is composite.

**Proof.** Two cases.

*Case 1 – `S` composite.*  
Algorithm outputs the whole array (size `n`).  
By Lemma&nbsp;1 this subset is valid and maximal.

*Case 2 – `S` prime.*  
Algorithm removes one odd element, outputs the remaining `n‑1` indices.  
By Lemma&nbsp;4 the sum of this subset is composite, so the subset is valid.  
By Lemma&nbsp;5 no subset of size `n` can be valid, therefore any valid
subset can have at most `n‑1` elements.  
Thus the algorithm’s subset of size `n‑1` is maximal.

Both cases prove the claim. ∎

#### 4.  Complexity Analysis  

For each test case  

* Computing the total sum: `O(n)`.
* Primality test of `S` ( ≤ 20000 ): trial division up to `√S` → `O(√S)` ≤ `O(150)`.
* Scanning for an odd element (if needed): `O(n)`.

Overall `O(n + √S)` time, at most a few hundred operations.
Memory consumption is `O(n)` for storing the array and indices.

```python
import sys
import math

def is_prime(x: int) -> bool:
    """return True if x is a prime number (x >= 2)"""
    if x < 2:
        return False
    if x == 2:
        return True
    if x % 2 == 0:
        return False
    r = int(math.isqrt(x))
    for d in range(3, r + 1, 2):
        if x % d == 0:
            return False
    return True

def solve() -> None:
    data = sys.stdin.read().strip().split()
    if not data:
        return
    it = iter(data)
    t = int(next(it))
    out_lines = []
    for _ in range(t):
        n = int(next(it))
        a = [int(next(it)) for _ in range(n)]
        total = sum(a)

        if not is_prime(total):
            # whole array works
            out_lines.append(str(n))
            out_lines.append(' '.join(str(i) for i in range(1, n + 1)))
        else:
            # total is prime -> remove any odd element
            odd_idx = -1
            for i, val in enumerate(a):
                if val % 2 == 1:
                    odd_idx = i          # 0‑based index
                    break
            # according to the proof such an index always exists
            remaining = [str(i) for i in range(1, n + 1) if i - 1 != odd_idx]
            out_lines.append(str(n - 1))
            out_lines.append(' '.join(remaining))

    sys.stdout.write('\n'.join(out_lines))

if __name__ == "__main__":
    solve()
```

**Solution Explanation**

For every test case we are given the watering plan of `n` days.

* The flower starts with height `h = 1`.
* While scanning the days from left to right we keep the previous day’s
  watering state (`prev`).
* For the current day `i`
    * if `a[i] == 1` (watered)  

        * if `prev == 1` the flower grows `+5`
        * otherwise it grows `+1`
    * else (`a[i] == 0`, not watered)  

        * if `prev == 0` we have two consecutive dry days → the flower dies.
          We can stop processing this test case and output `-1`.
* If we finish all days without dying we output the final height.

The algorithm is a single linear scan, `O(n)` time and `O(1)` extra memory.

---

#### Correctness Proof  

We prove that the algorithm outputs the required result for each test case.

*Lemma 1*  
When processing day `i` the variable `h` equals the flower’s height
after day `i‑1`.

*Proof.*  
Initially `h = 1`, which is the height before day 1, so the lemma holds for
`i = 1`.  
Assume it holds for some `i`.  
The algorithm updates `h` exactly according to the problem statement:

* if day `i` is watered and day `i‑1` was watered (`prev = 1`) it adds `5`,
  otherwise it adds `1`;
* if day `i` is not watered it adds nothing.

Thus after the update `h` equals the height after day `i`. ∎



*Lemma 2*  
If the algorithm prints `-1`, the flower must die according to the rules.

*Proof.*  
The algorithm prints `-1` only when it encounters a day `i` with
`a[i] = 0` and `prev = 0`.  
That means days `i‑1` and `i` are both not watered, i.e. two consecutive
dry days, which is exactly the death condition. ∎



*Lemma 3*  
If the algorithm finishes the scan without printing `-1`,
the flower never experiences two consecutive dry days.

*Proof.*  
The algorithm would have printed `-1` at the first occurrence of two
consecutive zeros. Since it never does, such a pair never appears. ∎



*Theorem*  
For each test case the algorithm outputs  

* `-1` iff the flower dies,  
* otherwise the exact height of the flower after `n` days.

*Proof.*  

*If the algorithm outputs `-1`*: by Lemma&nbsp;2 the flower dies, so the
required output is `-1`.

*If the algorithm outputs a number `H`*: by Lemma&nbsp;3 the flower never
dies. By Lemma&nbsp;1, after processing the last day `n` the variable
`h` equals the height after day `n`. The algorithm outputs this `h`,
hence the correct final height. ∎

```python
import sys

def solve() -> None:
    data = sys.stdin.read().strip().split()
    if not data:
        return
    it = iter(data)
    t = int(next(it))
    out_lines = []
    for _ in range(t):
        n = int(next(it))
        a = [int(next(it)) for _ in range(n)]

        height = 1          # initial height before day 1
        prev = 0            # pretend day 0 was not watered
        dead = False

        for i in range(n):
            cur = a[i]
            if cur == 1:
                if prev == 1:
                    height += 5
                else:
                    height += 1
            else:  # cur == 0
                if prev == 0:          # two dry days in a row
                    dead = True
                    break
            prev = cur

        out_lines.append("-1" if dead else str(height))

    sys.stdout.write("\n".join(out_lines))

if __name__ == "__main__":
    solve()
```

such that the sum of all consecutive integers from `l` to `r` equals `n`.

Consider the interval  

```
l = -n + 1
r =  n
```

*Number of terms*  

```
cnt = r - l + 1 = n - ( -n + 1 ) + 1 = 2n
```

*Sum of an arithmetic progression*  

```
sum = (first + last) * cnt / 2
    = ( (-n + 1) + n ) * (2n) / 2
    = 1 * (2n) / 2
    = n
```

So the interval `[-n+1 , n]` always gives the required sum.

--------------------------------------------------------------------

* `r = n ≤ 10^18` – given by the statement.
* `l = -n + 1 ≥ -10^18 + 1` – because `n ≤ 10^18`.  
  This is still inside the allowed range `[-10^18 , 10^18]`.
* `l < r` because `-n+1 < n` for every positive `n`.

Thus the pair is always valid.

For every test case we are given a board  

```

row 2 : s2[1 … n]
```

The character starts in cell `(1,1)` and has to reach `(2,n)`.  
In one move it may go to any of the 8 neighbour cells
(`|Δrow| ≤ 1 , |Δcol| ≤ 1 , not both zero`).  
The board has only two rows.

--------------------------------------------------------------------

#### 1.   Observation  

Because a move changes the column by at most `1`,
the character can never *skip* a column.
If for some column `c` **both** cells are traps (`s1[c] = s2[c] = '1'`)
the whole column `c` is forbidden – the character cannot stand in it
and also cannot jump over it.
Consequently the board is split into a left part (columns `< c`) and a
right part (columns `> c`) that are unreachable from each other.

The start cell is in column `1` (the leftmost column) and the target cell
is in column `n` (the rightmost column).  
If any column contains two traps, the left part and the right part are
separated, therefore the target can never be reached.

Conversely, if **no** column contains two traps, the character can always
walk column by column, possibly changing rows whenever a single trap is
present (diagonal moves allow that).  
Thus a path always exists.

So the whole problem reduces to a single check:

```
answer = "YES"  iff  there is no column i with s1[i] = s2[i] = '1'
```

##### Lemma 1  
If a column `c` satisfies `s1[c] = s2[c] = '1'` then no valid path can contain a
cell with column index `c`.

**Proof.** Both cells of column `c` are traps, therefore stepping on any of
them kills the character. Because a move changes the column by at most `1`,
the character cannot move from column `c-1` directly to column `c+1`.  
Hence column `c` cannot be entered at all. ∎



##### Lemma 2  
If a column `c` as in Lemma&nbsp;1 exists, there is no path from `(1,1)` to
`(2,n)`.

**Proof.** By Lemma&nbsp;1 the character can never be in column `c`.  
All columns `< c` are on the left side of the board, all columns `> c` on the
right side. The start cell `(1,1)` lies left of `c`, the target cell `(2,n)`
lies right of `c`. Since the character cannot cross column `c`, it cannot
reach the target. ∎



##### Lemma 3  
If no column contains two traps, a path from `(1,1)` to `(2,n)` exists.

**Proof.** Process columns from left to right.
Assume the character is currently in column `k` (starting with `k = 1`).
* If both cells of column `k+1` are safe, move straight to `(row, k+1)`.
* If exactly one of them is a trap, move diagonally to the safe cell of
  column `k+1` (possible because diagonal moves are allowed).
Thus the character can always advance to the next column while staying on a
safe cell. Repeating this for `k = 1 … n-1` reaches column `n`.  
The cell `(2,n)` is safe by the statement, therefore the constructed walk
ends at the required target. ∎



##### Theorem  
For each test case the algorithm outputs **YES** iff a valid path exists.

**Proof.**  

*If the algorithm outputs NO.*  
It found a column `c` with `s1[c] = s2[c] = '1'`.  
By Lemma&nbsp;2 no path exists. Hence the answer NO is correct.

*If the algorithm outputs YES.*  
It examined all columns and found none with two traps.  
By Lemma&nbsp;3 a path exists, therefore YES is correct.

Both directions hold, completing the proof. ∎

**Solution Explanation**

For every weight we have to decide

* which still unused weight we take,
* on which side (`L` – left, `R` – right) we put it.

After the `i`‑th move the total weight on the left side is `L_i`,
the total weight on the right side is `R_i`.
The balance must satisfy  

```
S[i] = 'L'   →   L_i  >  R_i          (the left side is heavier)
S[i] = 'R'   →   L_i  <  R_i          (the right side is heavier)
```

--------------------------------------------------------------------

#### 1.   Reformulation

Let  

```
diff = L – R                (difference after the already performed moves)
```

`diff` is `0` before the first move.

If we put a weight `w` on the left side, `diff` becomes `diff + w`.  
If we put it on the right side, `diff` becomes `diff – w`.

For the required sign `sgn` (`+1` for `L`, `-1` for `R`) we need

```
sgn * (new diff)  >  0
```

--------------------------------------------------------------------

#### 2.   Which weight can be used in one step ?

Assume the current value of `diff` is known and the required sign is
`sgn`.

*Put the weight on the left*  

```
new diff = diff + w
sgn * (diff + w) > 0      →      w  >  -sgn * diff
```

*Put the weight on the right*  

```
new diff = diff – w
sgn * (diff – w) > 0      →      w  <   sgn * diff
```

So for a fixed step we have at most two *intervals* of admissible
weights:

```
left side :   w  >  -sgn*diff
right side:   w  <   sgn*diff
```

If an interval is empty, the corresponding side cannot be used.

--------------------------------------------------------------------

#### 3.   Keeping the balance “easy” – greedy choice  

When both sides are possible we would like to keep `|diff|` as small as
possible – a small absolute difference is easier to change later.
Therefore we try both admissible sides, compute the resulting
`|new diff|` and take the one with the smaller absolute value
(if only one side is possible we have no choice).

--------------------------------------------------------------------

#### 4.   Data structure – we need the smallest weight **greater**
than a value and the largest weight **smaller** than a value,
and we have to delete used weights.

All weights are distinct, therefore we can sort them once,
compress them to indices `0 … N‑1` and work with a
**Fenwick tree (Binary Indexed Tree)** that stores for every index
whether the weight is still unused (`1`) or already taken (`0`).

The Fenwick tree supports

* `add(pos, delta)` – change the stored value (`O(log N)`);
* `pref(pos)` – number of still unused weights with index `≤ pos`
  (`O(log N)`);
* `kth(k)` – the index of the *k‑th* still unused weight
  (`O(log N)`, binary lifting).

With the sorted array `sorted_weights` we can

* find the first index `idx` with `sorted_weights[idx] > x`
  (`bisect_right`);
* ask the Fenwick tree whether there is any still unused weight
  with index `≥ idx` – if yes, the wanted weight is the
  `(pref(idx‑1)+1)`‑st unused weight, obtained by `kth`;
* analogously for the largest weight `< x`.

All operations are `O(log N)`, therefore the whole algorithm works in
`O(N log N)`.

--------------------------------------------------------------------

#### 5.   Algorithm
```
read N, array A, string S
sorted_weights = sorted(A)
compress every weight → its index in sorted_weights
Fenwick tree FT, initially FT[i] = 1 for all i

diff = 0
answer = empty list

for i = 0 … N-1
        s = S[i]                     # required side after this move
        sgn = +1 if s == 'L' else -1

        candidates = empty list

        # ----- try to put the weight on the left side -----
        # condition: w > -sgn*diff
        low = -sgn*diff
        idx = bisect_right(sorted_weights, low)          # first > low
        if idx < N and FT.pref(N-1) - FT.pref(idx-1) > 0
                kth = FT.pref(idx-1) + 1
                pos = FT.kth(kth)                        # index of weight
                w   = sorted_weights[pos]
                new_diff = diff + w
                candidates.append( (abs(new_diff), w, 'L', pos) )

        # ----- try to put the weight on the right side -----
        # condition: w <  sgn*diff
        high = sgn*diff
        idx2 = bisect_left(sorted_weights, high) - 1    # last < high
        if idx2 >= 0 and FT.pref(idx2) > 0
                kth = FT.pref(idx2)                     # last present ≤ idx2
                pos = FT.kth(kth)
                w   = sorted_weights[pos]
                new_diff = diff - w
                candidates.append( (abs(new_diff), w, 'R', pos) )

        if candidates is empty
                print -1 and exit

        choose the candidate with minimal first component (|new diff|)
        (abs_val, w, side, pos) = that candidate

        answer.append( (w, side) )
        FT.add(pos, -1)                # weight is now used
        diff = diff + w   if side == 'L' else diff - w
end for

print the pairs stored in answer, one per line
```

We prove that the algorithm prints a correct sequence
or correctly reports that none exists.

---

##### Lemma 1  
For a fixed step with current difference `diff` and required sign
`sgn` the algorithm enumerates **all** still unused weights that can be
placed on the left side **and** all that can be placed on the right side.

**Proof.**  
The condition for the left side is `w > -sgn*diff`.  
`bisect_right` finds the first index `idx` with `sorted_weights[idx] > -sgn*diff`.  
All admissible weights have an index `≥ idx`.  
The Fenwick tree tells whether at least one such weight is still unused.
If yes, the first unused one is the `(pref(idx‑1)+1)`‑st unused weight,
found by `kth`.  
Thus the algorithm obtains the smallest admissible weight for the left
side; any other admissible weight would have a larger index and is also
present in the tree, therefore it would also be found if we asked for it.
The same argument with `bisect_left` works for the right side.
∎



##### Lemma 2  
If the algorithm chooses a candidate, after performing the move the
balance satisfies the required sign of the current character of `S`.

**Proof.**  
The candidate is produced only after checking the corresponding
inequality (`w > -sgn*diff` for a left move, `w < sgn*diff` for a right
move).  
These inequalities are exactly the algebraic reformulation of
`sgn * (new diff) > 0`.  
Therefore the new difference has the correct sign. ∎



##### Lemma 3  
If at some step the algorithm reports “no candidate”, then **no**
solution for the whole instance exists.

**Proof.**  
Assume a solution exists.
Consider the first step where the algorithm fails.
All weights that could be placed on the left side satisfy
`w > -sgn*diff`; all that could be placed on the right side satisfy
`w < sgn*diff`.  
By Lemma&nbsp;1 the algorithm would have found every still unused weight
fulfilling one of the two conditions.
Because it found none, no unused weight can be placed on either side
while keeping the required sign – contradicting the existence of a
solution. ∎



##### Lemma 4  
If the algorithm finishes all `N` steps, the produced sequence is a
valid solution.

**Proof.**  
Induction over the steps.

*Base.* Before the first step `diff = 0`.  
The algorithm selects a weight according to Lemma&nbsp;1,
the side according to Lemma&nbsp;2, therefore the first requirement of
`S` is satisfied.

*Induction step.* Assume after step `i‑1` the balance satisfies the
first `i‑1` characters of `S`.  
At step `i` the algorithm again enumerates all admissible weights
(Lemma&nbsp;1) and picks one that keeps the required sign
(Lemma&nbsp;2).  
Thus after step `i` the first `i` characters are satisfied.
∎



##### Lemma 5  
If a solution exists, the algorithm never gets stuck (i.e. it never
fails to find a candidate).

**Proof.**  
Assume a solution exists and the algorithm gets stuck at step `k`.
By Lemma&nbsp;3 this would mean that no unused weight can be placed on
either side while keeping the required sign – impossible because the
existing solution uses some still unused weight at step `k` and
obviously respects the sign. Contradiction. ∎



##### Theorem  
The algorithm outputs a sequence of moves that satisfies all
requirements of the string `S` **iff** such a sequence exists.
If no sequence exists it outputs `-1`.

**Proof.**  

*If the algorithm outputs a sequence* – by Lemma&nbsp;4 it is a valid
solution.

*If a solution exists* – by Lemma&nbsp;5 the algorithm never gets
stuck, therefore it processes all `N` steps and outputs a sequence,
which by the first part is a valid solution.

*If the algorithm prints `-1`* – it got stuck, and by Lemma&nbsp;3
no solution exists. ∎

```python
import sys
import bisect

# ------------------------------------------------------------
# Fenwick tree (Binary Indexed Tree) with kth order statistic
# ------------------------------------------------------------
class Fenwick:
    def __init__(self, n):
        self.n = n
        self.bit = [0] * (n + 1)          # 1‑based

    def add(self, idx, delta):
        """add delta at position idx (0‑based)"""
        i = idx + 1
        while i <= self.n:
            self.bit[i] += delta
            i += i & -i

    def pref(self, idx):
        """sum of positions 0 .. idx (inclusive), idx may be -1 → 0"""
        if idx < 0:
            return 0
        i = idx + 1
        s = 0
        while i:
            s += self.bit[i]
            i -= i & -i
        return s

    def total(self):
        return self.pref(self.n - 1)

    # smallest index such that prefix sum >= k (k is 1‑based)
    def kth(self, k):
        idx = 0
        bitmask = 1 << (self.n.bit_length())   # largest power of two >= n
        while bitmask:
            nxt = idx + bitmask
            if nxt <= self.n and self.bit[nxt] < k:
                k -= self.bit[nxt]
                idx = nxt
            bitmask >>= 1
        return idx          # 0‑based index (because idx is the last position with sum < original k)

# ------------------------------------------------------------
def solve() -> None:
    data = sys.stdin.read().strip().split()
    if not data:
        return
    it = iter(data)
    n = int(next(it))
    A = [int(next(it)) for _ in range(n)]
    S = next(it).strip()

    sorted_w = sorted(A)
    # map weight -> its index in sorted_w (unique because all distinct)
    pos_of = {w: i for i, w in enumerate(sorted_w)}

    ft = Fenwick(n)
    for i in range(n):
        ft.add(i, 1)                     # all weights are initially unused

    diff = 0
    answer = []

    for ch in S:
        sgn = 1 if ch == 'L' else -1
        candidates = []

        # ----- try left side (weight added) -----
        low = -sgn * diff                # need w > low
        idx = bisect.bisect_right(sorted_w, low)   # first index with weight > low
        if idx < n:
            # is there any unused weight with index >= idx ?
            if ft.total() - ft.pref(idx - 1) > 0:
                kth = ft.pref(idx - 1) + 1
                pos = ft.kth(kth)
                w = sorted_w[pos]
                new_diff = diff + w
                candidates.append((abs(new_diff), w, 'L', pos))

        # ----- try right side (weight subtracted) -----
        high = sgn * diff                # need w < high
        idx2 = bisect.bisect_left(sorted_w, high) - 1   # last index with weight < high
        if idx2 >= 0:
            cnt = ft.pref(idx2)
            if cnt > 0:
                pos = ft.kth(cnt)       # the cnt‑th unused weight = last ≤ idx2
                w = sorted_w[pos]
                new_diff = diff - w
                candidates.append((abs(new_diff), w, 'R', pos))

        if not candidates:
            print(-1)
            return

        # choose the candidate with smallest |new diff|
        candidates.sort()
        _, w, side, pos = candidates[0]

        answer.append((w, side))
        ft.add(pos, -1)                 # remove this weight
        diff = diff + w if side == 'L' else diff - w

    out_lines = [f"{w} {side}" for w, side in answer]
    sys.stdout.write("\n".join(out_lines))

# ------------------------------------------------------------
if __name__ == "__main__":
    solve()
```

**Solution Explanation**

For one operation we choose exactly `k` different positions  

```
i1 < i2 < … < ik
```

and compute  

```
x = a[i1] & a[i2] & … & a[ik]                (bitwise AND)
```

and replace every chosen element by `a[ij] – x`.

--------------------------------------------------------------------

#### 1.   Observations about one bit  

Look at a single binary bit `b` ( `0 ≤ b < 30` ).

*`cnt[b]`* – how many array elements have this bit equal to `1`.

During one operation the bit `b` of the chosen `k` numbers is

* `1` in `x` **iff** all `k` numbers have this bit equal to `1`.
* If the bit is `1` in `x`, after subtraction the bit becomes `0`
  in all those `k` numbers.
* If the bit is `0` in `x`, nothing changes for this bit.

Therefore one operation **decreases** `cnt[b]` either by `0` or by `k`,
never by any other amount.

--------------------------------------------------------------------

#### 2.   Necessary condition for a feasible `k`

At the end all bits are `0`, i.e. `cnt[b] = 0` for every `b`.
Consequently the initial value `cnt[b]` must be a multiple of `k`.

```
k | cnt[b]   for every bit b
```

Let  

```
G = gcd of all cnt[b] (ignoring cnt[b] = 0)
```

Then the condition is equivalent to  

```
k divides G
```

If the whole array is already zero, every `cnt[b] = 0` and we define
`G = 0`. In this case **any** `k (1 … n)` works.

--------------------------------------------------------------------

#### 3.   Sufficient condition  

Assume `k` divides every `cnt[b]`.  
We show that we can always finish the array.

Take any moment while some element is still non‑zero.
Because each `cnt[b]` is a multiple of `k`, the total number of
still non‑zero elements is also a multiple of `k`.  
Pick any `k` of them (possible because the number is ≥ `k`).

Among these `k` numbers there exists at least one bit that is `1` in
**all** of them – otherwise the AND would be `0` and the operation
would do nothing, contradicting the fact that at least one chosen
element is non‑zero.  
Hence the AND `x` is non‑zero, and after the operation the chosen
`k` numbers lose that common bit, i.e. `cnt[that_bit]` is reduced by
exactly `k`. All other bits are reduced either by `0` or by `k` as
well, therefore the divisibility by `k` is preserved.

Repeating the step we finally make every `cnt[b]` equal to `0`,
hence the whole array becomes zero.

Thus the condition “`k` divides `G`” is also sufficient.

--------------------------------------------------------------------

#### 4.   Result  

*If the array is all zeros* → answer is `1,2,…,n`.

*Otherwise*  

```
G = gcd of all cnt[b] (cnt[b] > 0)
answer = all k (1 ≤ k ≤ n) such that G % k == 0
```

#### 6.   Correctness Proof  

We prove that the algorithm prints exactly the feasible `k`.

---

##### Lemma 1  
During any elimination operation the number of ones of a fixed bit `b`
decreases by either `0` or `k`.

**Proof.**  
`x` contains bit `b` equal to `1` iff every chosen element has this bit
`1`. In that case all `k` chosen elements lose the bit after subtraction,
so the count drops by `k`. Otherwise the bit of `x` is `0` and nothing
changes for this bit. ∎



##### Lemma 2  
If it is possible to make the whole array zero using operations of size
`k`, then `k` divides `cnt[b]` for every bit `b`.

**Proof.**  
Initially the count of ones of bit `b` equals `cnt[b]`. By Lemma&nbsp;1 each
operation changes this count by a multiple of `k`. The final count is
`0`. Hence `cnt[b]` must be a multiple of `k`. ∎



##### Lemma 3  
Let `G` be the gcd of all positive `cnt[b]`.  
If `k` divides `G` then the array can be reduced to all zeros using
operations of size `k`.

**Proof.**  
Because `k | G`, `k` divides every `cnt[b]`.  
While some element is non‑zero, the total number of non‑zero elements is
a multiple of `k` (sum of the `cnt[b]` over all bits, each multiple of `k`).
Pick any `k` of them.  
If their bitwise AND were `0`, all chosen numbers would be unchanged,
contradicting that at least one of them is non‑zero.  
Thus the AND contains a bit that is `1` in all chosen numbers; after the
operation that bit disappears from exactly those `k` numbers, decreasing
its `cnt` by `k`. All other bits stay multiples of `k`.  
Repeating the step eventually makes every `cnt[b]` zero, i.e. the whole
array zero. ∎



##### Lemma 4  
If the whole array is already zero (`G = 0`), any `k (1 … n)` is feasible.

**Proof.** No operation is needed, so the requirement is trivially met. ∎



##### Theorem  
For each test case the algorithm outputs precisely the integers `k`
(`1 ≤ k ≤ n`) for which the array can be turned into all zeros by a
finite sequence of elimination operations of size `k`.

**Proof.**  

*Case 1 – all array elements are zero.*  
The algorithm outputs `1 … n`. By Lemma&nbsp;4 each of them is feasible,
and obviously no other integer exists, so the output is exact.

*Case 2 – at least one element is non‑zero.*  
The algorithm computes `G` as the gcd of the positive `cnt[b]` and
outputs all `k` dividing `G`.  

*Soundness* – any printed `k` divides `G`, therefore divides every
`cnt[b]`. By Lemma&nbsp;2 such a `k` is necessary, and by Lemma&nbsp;3 it
is sufficient, hence feasible.

*Completeness* – let `k` be any feasible size. By Lemma&nbsp;2 it divides
every `cnt[b]`, consequently it divides their gcd `G`. Therefore `k`
appears in the list produced by the algorithm.

Both directions hold, so the set printed by the algorithm equals the
set of feasible `k`. ∎

**Solution Explanation**

For a sequence `b` we need  

```
b[i] ≤ i          (1‑based index)
```

Only **insertions** are allowed, we never delete or change an existing element.
When we insert a number before some element, the element’s final index becomes
larger, therefore it becomes easier to satisfy the condition.

--------------------------------------------------------------------

#### 1.   Greedy view

Process the original elements from left to right.
Assume we have already processed the first `i‑1` original numbers and inserted
some extra numbers.
Let  

```
len = current length of the whole sequence (original + inserted)
```

The next original element `a[i]` will be placed at position `len+1`
(if we do not insert anything before it).

*If `a[i] ≤ len+1`* – the condition is already satisfied, we just put the
element there.

*If `a[i] > len+1`* – we have to increase its index.
The cheapest way is to insert exactly  

```
need = a[i] - (len+1)
```

new numbers (any positive values, e.g. `1`) right before `a[i]`.
After those insertions the length becomes `len + need`,
so the element will finally be at position `len+need+1 = a[i]`,
which fulfills `a[i] ≤ position`.

No other strategy can use fewer insertions for this element,
because each insertion can increase the index by **at most one**.
Therefore the greedy choice is optimal for the current element,
and it never harms the future ones – a larger `len` can only help later
elements.

--------------------------------------------------------------------

#### 2.   Algorithm
For each test case  

```
ans = 0                # total insertions
len = 0                # current length of the sequence

for x in original array:
        if x > len + 1:
                need = x - (len + 1)
                ans += need
                len += need          # inserted numbers
        len += 1                    # place the original element
output ans
```

--------------------------------------------------------------------

#### 3.   Correctness Proof  

We prove that the algorithm returns the minimum possible number of insertions.

---

##### Lemma 1  
When processing an original element `x` with current length `len`,
any feasible final sequence must contain at least  

```
max(0, x - (len+1))
```

insertions before this element.

**Proof.**  
In the final sequence the element will be at some position `p ≥ len+1`
(because at most `len` elements are already placed before it).
The condition requires `x ≤ p`, i.e. `p ≥ x`.  
Thus we need `p ≥ max(len+1, x)`.  
If `x ≤ len+1` we can take `p = len+1` and need no insertion.
Otherwise `p` must be at least `x`, which is `x-(len+1)` positions larger
than `len+1`. Each insertion can increase the position by at most one,
hence at least `x-(len+1)` insertions are necessary. ∎



##### Lemma 2  
The algorithm inserts exactly the lower bound of Lemma&nbsp;1 for every
original element.

**Proof.**  
The algorithm computes `need = max(0, x-(len+1))` and inserts that many
numbers before `x`. So the number of insertions equals the lower bound. ∎



##### Lemma 3  
After the algorithm has processed the first `i` original elements,
the current length `len` equals the smallest possible length of any
sequence that satisfies the condition for those `i` elements.

**Proof by induction on `i`.**

*Base `i = 0`*: `len = 0`, obviously minimal.

*Induction step*: assume the claim holds for `i‑1`.  
For element `a[i]` the algorithm adds the minimal required insertions
(Lemma&nbsp;2). Any feasible sequence must contain at least those insertions
(Lemma&nbsp;1). Adding the element itself increases the length by one in any
case. Therefore after processing `i` elements the algorithm’s length is
minimal. ∎



##### Lemma 4  
The total number `ans` produced by the algorithm is feasible
(i.e. the final sequence satisfies `b_j ≤ j`).

**Proof.**  
By construction each original element `a[i]` is placed at position
`max(len_before+1, a[i])`.  
If `a[i] ≤ len_before+1` it is placed at `len_before+1` and the condition holds.
Otherwise we inserted exactly `a[i]-(len_before+1)` numbers, so its final
position becomes `a[i]`. In both cases `a[i] ≤ its position`.  
All inserted numbers are positive, therefore also satisfy the condition.
Thus the whole final sequence fulfills the requirement. ∎



##### Theorem  
For every test case the algorithm outputs the minimum possible number of
operations.

**Proof.**  
Let `Ans_opt` be the optimal (minimum) number of insertions.
From Lemma&nbsp;1 each original element needs at least the number of insertions
used by the algorithm, therefore `Ans_opt ≥ ans`.  
From Lemma&nbsp;4 the algorithm’s `ans` insertions are sufficient, so
`Ans_opt ≤ ans`.  
Consequently `Ans_opt = ans`. ∎

**Solution Explanation**

For a string `s` (only letters `a` and `b`)  

* `AB(s)` – number of substrings `"ab"`  
* `BA(s)` – number of substrings `"ba"`

For every adjacent pair `s[i] s[i+1]`

```
pair = "ab" → contributes +1 to (AB - BA)
pair = "ba" → contributes -1 to (AB - BA)
pair = "aa" or "bb" → contributes 0
```

Define  

```
val('a') = 1 ,  val('b') = 0
```

Then for a pair `s[i] s[i+1]`

```
contribution = val(s[i]) - val(s[i+1])
```

Therefore  

```
AB(s) - BA(s) = Σ (val(s[i]) - val(s[i+1]))   (i = 0 … n-2)
               = val(s[0]) - val(s[n-1])
```

`val(s[0])` is `1` iff the first character is `a`, otherwise `0`.  
`val(s[n-1])` is `1` iff the last character is `a`, otherwise `0`.

Hence  

```
diff = AB(s) - BA(s) = (s[0] == 'a') - (s[-1] == 'a')
```

`diff` can only be `-1 , 0 , 1`.

* `diff = 0` → `AB(s) = BA(s)` already, no change needed.
* `diff = 1` → first character is `a`, last is `b`.  
  Changing **any** one of them (first → `b` or last → `a`) makes the
  two indicators equal, so `diff` becomes `0`. One step is enough.
* `diff = -1` → first is `b`, last is `a`.  
  Symmetrically, change first → `a` or last → `b`. One step is enough.

Thus the minimum number of modifications is  

```
0  if s[0] == s[-1]
1  otherwise
```

Any string obtained by changing one of the two end characters to the
other end character is a correct answer.

#### Algorithm
For each test case  

```
if s[0] == s[-1]:
        answer = s                     # already balanced
else:
        # change the first character to the last one (any choice works)
        answer = s[-1] + s[1:]          # replace first char
output answer
```

The string length is at most `100`, so this is O(n) time and O(1) extra
memory.

We prove that the algorithm outputs a string `t` with `AB(t)=BA(t)` and
uses the minimum possible number of changes.

---

**Lemma 1**  
For any string `x` consisting of `a` and `b`

```
AB(x) - BA(x) = (x[0] == 'a') - (x[-1] == 'a')
```

*Proof.*  
For each adjacent pair `x[i]x[i+1]` define `c(i) = 1` if the pair is
`"ab"`, `c(i) = -1` if it is `"ba"`, otherwise `0`.  
`c(i) = val(x[i]) - val(x[i+1])` with `val('a')=1, val('b')=0`. Summing
over all `i` telescopes to `val(x[0]) - val(x[-1])`, i.e. the right‑hand
side. ∎



**Lemma 2**  
If the first and last characters of a string are equal, then
`AB = BA`.

*Proof.*  
When they are equal, the right‑hand side of Lemma&nbsp;1 is `0`,
hence `AB-BA = 0`. ∎



**Lemma 3**  
If the first and last characters differ, changing **any** one of them
to the other character makes `AB = BA`.

*Proof.*  
Assume `s[0] ≠ s[-1]`. After changing the first character to `s[-1]`,
the new first character equals the last one, so by Lemma&nbsp;2 the new
string satisfies `AB = BA`. The same holds if we change the last
character instead. ∎



**Lemma 4**  
When `s[0] ≠ s[-1]`, at least one modification is necessary.

*Proof.*  
By Lemma&nbsp;1 the original string has `AB-BA = ±1`. Any string with
`AB = BA` has difference `0`. Changing a single character can alter the
value of `(s[0]=='a')` or `(s[-1]=='a')` by at most `1`, therefore at
least one change is required. ∎



**Theorem**  
For every test case the algorithm outputs a string `t` such that
`AB(t)=BA(t)` and the number of changed positions is minimal.

*Proof.*  

*Case 1:* `s[0] == s[-1]`.  
The algorithm returns `s` unchanged. By Lemma&nbsp;2 the condition
holds, and zero changes are obviously minimal.

*Case 2:* `s[0] != s[-1]`.  
The algorithm changes the first character to the last one, i.e. exactly
one position. By Lemma&nbsp;3 the resulting string satisfies
`AB = BA`. By Lemma&nbsp;4 any solution needs at least one change, so
the algorithm is optimal.

Thus in all cases the algorithm is correct and optimal. ∎

While typing we start on the first letter of `s` without any cost.

The total time is therefore the sum of the distances between every two
consecutive letters of `s`.

--------------------------------------------------------------------

#### Algorithm
For each test case  

1. Build an array `pos[26]` where `pos[c]` is the index of character `c`
   in the keyboard string `k`.  
   (e.g. `pos['a'] = 0` if `'a'` is the leftmost key.)
2. Initialise `answer = 0`.
3. For every `i = 1 … len(s)-1`  

   ```
   answer += abs( pos[ s[i] ] - pos[ s[i-1] ] )
   ```

4. Output `answer`.

--------------------------------------------------------------------

#### Correctness Proof  

We prove that the algorithm outputs the exact time needed to type `s`.

*Lemma 1*  
For any character `c` the algorithm stores in `pos[c]` its true position on
the keyboard.

*Proof.*  
`k` is a permutation of the alphabet. While scanning `k` once,
the algorithm records the index at which each character occurs.
Thus `pos[c]` equals the index of `c` in `k`, i.e. its position. ∎



*Lemma 2*  
For each `i (1 ≤ i < |s|)` the term  
`abs(pos[s[i]] - pos[s[i-1]])` computed by the algorithm equals the time
required to move the hand from the key of `s[i-1]` to the key of `s[i]`.

*Proof.*  
By Lemma&nbsp;1 `pos[s[i-1]]` and `pos[s[i]]` are exactly the positions of the two
keys. The definition of the movement cost is the absolute difference of the
positions, which is precisely the term above. ∎



*Lemma 3*  
`answer` after the loop equals the total time needed to type the whole word
`s`.

*Proof.*  
The total time is the sum of the movement costs between every pair of
consecutive letters of `s`. By Lemma&nbsp;2 each loop iteration adds exactly the
cost of one such movement, and the loop iterates over all `|s|-1` pairs.
Therefore the final `answer` is the required total sum. ∎



*Theorem*  
For each test case the algorithm outputs the minimal (and only) time needed to
type the word `s` on the given keyboard.

*Proof.*  
The problem statement defines the typing time uniquely as the sum of the
movement costs between consecutive letters, with no cost for the first
letter. By Lemma&nbsp;3 the algorithm computes exactly this sum, hence the
output is the correct typing time. ∎

* `'='` if `N1 = N2`



--------------------------------------------------------------------

#### 1.  Observations  

* Let `d1 = number of decimal digits of x1` (`d1 = len(str(x1))`), similarly `d2`.
* The total length (number of digits) of the whole numbers is  

```
len1 = d1 + p1          # digits of N1
len2 = d2 + p2          # digits of N2
```

If `len1` and `len2` are different, the longer one is larger – no further work is needed.

* If the lengths are equal, the two numbers have the same amount of digits.
  Then the only possible difference is in the *most significant* part,
  i.e. in the `x` values after we have aligned the trailing zeros.

  Because the total lengths are equal  

```
d1 + p1 = d2 + p2   →   |p1 - p2| = |d2 - d1|
```

  The difference of the exponents is at most the difference of the
  digit‑counts of the two `x` values.
  `x ≤ 10^6` → `d ≤ 7`, therefore  

```
|p1 - p2| ≤ 7
```

  So after making the exponents equal we have to compare two integers
  that are at most `10^6·10^7 = 10^13`, easily fitting into Python’s
  arbitrary‑precision `int`.

--------------------------------------------------------------------

#### 2.  Algorithm
For each test case  

```
d1 = digits(x1)          # len(str(x1))
d2 = digits(x2)

len1 = d1 + p1
len2 = d2 + p2

if len1 < len2 : answer = '<'
elif len1 > len2 : answer = '>'
else:                     # same total length
        # make the exponents equal
        if p1 > p2:
                x1 *= 10**(p1-p2)      # p1-p2 ≤ 7
        elif p2 > p1:
                x2 *= 10**(p2-p1)

        # now compare the (still small) integers
        if x1 < x2 : answer = '<'
        elif x1 > x2 : answer = '>'
        else : answer = '='
output answer
```

--------------------------------------------------------------------

#### 3.  Correctness Proof  

We prove that the algorithm outputs the correct relation for every test case.

---

##### Lemma 1  
If `len1 ≠ len2` then the number with larger `len` is larger.

**Proof.**  
Both numbers are positive and written in decimal without leading zeros.
The one with more digits has a non‑zero most significant digit at a higher
position, therefore it is larger. ∎



##### Lemma 2  
If `len1 = len2` then `|p1-p2| ≤ 7`.

**Proof.**  
`len1 = d1 + p1` and `len2 = d2 + p2`. Equality gives `d1-p1 = d2-p2`,
hence `|p1-p2| = |d1-d2|`.  
`x ≤ 10^6` → `1 ≤ d ≤ 7`. Therefore `|d1-d2| ≤ 7`. ∎



##### Lemma 3  
Assume `len1 = len2`. After multiplying the `x` belonging to the smaller
exponent by `10^{|p1-p2|}` the two numbers become equal to the original
numbers and have the **same** exponent.

**Proof.**  
Suppose `p1 > p2`. Original numbers:

```
N1 = x1·10^{p1}
N2 = x2·10^{p2}
```

Multiply `x2` by `10^{p1-p2}`:

```
x2' = x2·10^{p1-p2}
N2' = x2'·10^{p2} = x2·10^{p1-p2}·10^{p2} = x2·10^{p1} = N2
```

Thus `N2' = N2` and its exponent is now `p1`. The same reasoning works for
`p2 > p1`. ∎



##### Lemma 4  
When `len1 = len2` the algorithm compares the two numbers correctly.

**Proof.**  
By Lemma&nbsp;3 the algorithm transforms the pair `(x1,p1),(x2,p2)` into
`(x1',p) , (x2',p)` where `p = max(p1,p2)` and the represented numbers are
unchanged. Because the exponents are equal, the order of the whole numbers
is exactly the order of the integer parts `x1'` and `x2'`. The algorithm
compares those integers, therefore it yields the correct relation. ∎



##### Theorem  
For every test case the algorithm outputs  

* `'<'` iff the first described number is smaller,
* `'>'` iff it is larger,
* `'='` iff they are equal.

**Proof.**  
Two cases.

*If `len1 ≠ len2`*: by Lemma&nbsp;1 the algorithm’s decision based on the
lengths is correct.

*If `len1 = len2`*: by Lemma&nbsp;4 the comparison after equalising the
exponents is correct.

Both cases cover all possibilities, hence the theorem holds. ∎

*Computing digit counts* – `O(1)` (conversion to string of at most 7 characters).  
*All other operations* – constant time, because the exponent difference is
bounded by 7, so the integer multiplication `10**k` and the final integer
comparison are `O(1)`.

Therefore for `t` test cases  

```
Time   : O(t)
```

For one test case we are given  

* `n` – number of towers  
* `a1 … an` – their heights  

In one operation we may move a single block from any tower `i` to any other tower `j`
(`i ≠ j`).  
The total number of blocks

```
S = a1 + a2 + … + an
```

never changes.

The *ugliness* of a configuration is  

```
max height – min height
```

We may perform any number of operations, i.e. we may redistribute the `S` blocks
arbitrarily among the `n` towers, as long as every tower height stays an integer
(and may become `0`).  
The task is to obtain the smallest possible ugliness.



--------------------------------------------------------------------

#### 1.   What is the optimal distribution?

If we could give every tower the same height, ugliness would be `0`.  
That is possible **iff** `S` is divisible by `n` :

```
S = n * (S / n)   →   each tower gets S / n blocks
```

Otherwise `S = n·q + r` with `0 < r < n` (`q = floor(S/n)`, `r = S mod n`).

The best we can do is to keep the heights as close as possible:

* `r` towers receive `q+1` blocks,
* the remaining `n‑r` towers receive `q` blocks.

Then  

```
max = q+1 ,   min = q   →   ugliness = 1
```

No other integer distribution can have a smaller difference,
because at least one tower must contain `⌈S/n⌉ = q+1` blocks
and at least one tower must contain `⌊S/n⌋ = q` blocks.

Hence the answer for a test case is

```
0 , if S % n == 0
1 , otherwise
```

--------------------------------------------------------------------

#### 2.   Algorithm
For each test case  

1. read `n` and the list `a`
2. compute `S = sum(a)`
3. output `0` if `S % n == 0` else `1`

--------------------------------------------------------------------

#### 3.   Correctness Proof  

We prove that the algorithm outputs the minimum possible ugliness.

---

**Lemma 1**  
If `S % n == 0` then ugliness `0` is achievable.

*Proof.*  
`S = n·k` for `k = S/n`. Give every tower exactly `k` blocks.
All heights are equal, so `max - min = 0`. ∎



**Lemma 2**  
If `S % n = r` with `0 < r < n` then any distribution of the `S` blocks has
`max - min ≥ 1`.

*Proof.*  
Let `q = ⌊S/n⌋`. Because `S = n·q + r` and `r > 0`,
the average height is `q + r/n`, which is strictly larger than `q`.
Therefore at least one tower must have height `≥ q+1` (otherwise the total
would be at most `n·q`).  
Similarly, because the total is less than `n·(q+1)`,
at least one tower must have height `≤ q`.  
Consequently `max ≥ q+1` and `min ≤ q`, giving `max - min ≥ 1`. ∎



**Lemma 3**  
If `S % n = r` with `0 < r < n` then ugliness `1` is achievable.

*Proof.*  
Construct the distribution described above:
* `r` towers receive `q+1` blocks,
* `n‑r` towers receive `q` blocks.  

The sum is `r·(q+1) + (n‑r)·q = n·q + r = S`.  
Here `max = q+1`, `min = q`, so `max - min = 1`. ∎



**Theorem**  
For each test case the algorithm outputs the minimum possible ugliness.

*Proof.*  
Two cases.

*Case 1:* `S % n == 0`.  
By Lemma&nbsp;1 ugliness `0` can be achieved, and no ugliness can be negative,
so `0` is optimal. The algorithm outputs `0`.

*Case 2:* `S % n != 0`.  
Lemma&nbsp;2 shows any configuration has ugliness at least `1`.  
Lemma&nbsp;3 shows ugliness `1` is attainable.  
Thus the optimum equals `1`, which the algorithm outputs.

In both cases the algorithm returns the optimal value. ∎

For each test case we only compute the sum of `n` numbers.

```
Time   :  O(n)
Memory :  O(1)   (apart from the input list)
```

For every test case we are given  

* a string `S` ( `1 ≤ |S| ≤ 100` )  
* a string `T` – a permutation of `"abc"` (there are only six possibilities)

We may permute the characters of `S` arbitrarily.
Among all permutations we have to output the **lexicographically smallest**
one that **does not contain `T` as a subsequence**.



--------------------------------------------------------------------

#### 1.  Observations  

* If we write the characters of `S` in increasing order (the usual
  *sorted* string) we obtain the globally smallest permutation of `S`.
* The sorted string can contain `T` only if `T = "abc"` :

  * sorted order = `aaa…bbb…ccc…ddd…`
  * for any other permutation of `"abc"` the required order of the three
    letters is impossible, because at least one of the letters would have
    to appear **before** a smaller one, which never happens in a sorted
    string.

* Even for `T = "abc"` the sorted string is safe when at least one of the
  letters `a , b , c` is missing – then `"abc"` can never be a subsequence.

So the only *dangerous* situation is  

```
T == "abc"   and   cnt['a']>0 , cnt['b']>0 , cnt['c']>0
```

--------------------------------------------------------------------

#### 2.  What to do in the dangerous situation?  

All `a` must stay at the very beginning – any other character would be
lexicographically larger.

After the block of `a` we are not allowed to put a `b` **before** a `c`,
otherwise the subsequence `a…b…c` would appear.
The smallest possible character that can be placed next is therefore `c`
(and all `c` have to be placed before the first `b`).

Consequently the minimal safe order is  

```
a…a   c…c   b…b   (all remaining letters in normal order)
```

All letters `d … z` can stay sorted, they never influence the existence of
`"abc"`.

--------------------------------------------------------------------

#### 3.  Algorithm
For each test case  

```
count occurrences of every letter in S   (array cnt[26])

if T == "abc" and cnt['a']>0 and cnt['b']>0 and cnt['c']>0:
        answer =  'a' * cnt['a'] +
                  'c' * cnt['c'] +
                  'b' * cnt['b'] +
                  for ch from 'd' to 'z':
                         ch * cnt[ch]
else:
        answer = for ch from 'a' to 'z':
                     ch * cnt[ch]          # completely sorted
output answer
```

--------------------------------------------------------------------

#### 4.  Correctness Proof  

We prove that the algorithm outputs exactly the required string.

---

##### Lemma 1  
If `T` is not `"abc"` then the sorted string of `S` does **not** contain `T`
as a subsequence.

**Proof.**  
In the sorted string all `a` come before all `b`, which come before all `c`.
Any permutation of `"abc"` different from `"abc"` needs at least one of the
relations `b` before `a`, `c` before `b`, or `c` before `a`.  
None of these relations holds in the sorted string, therefore the
permutation cannot be a subsequence. ∎



##### Lemma 2  
Assume `T = "abc"` and `cnt['a'],cnt['b'],cnt['c'] > 0`.  
The string  

```
A = a…a c…c b…b (rest sorted)
```

does **not** contain `"abc"` as a subsequence.

**Proof.**  
All `a` are placed before any `c` and any `b`.  
All `c` are placed before any `b`.  
Thus after an `a` the first possible occurrence of `b` is **after** every
`c`. Consequently there is no `a` – `b` – `c` in this order, i.e. `"abc"`
is not a subsequence of `A`. ∎



##### Lemma 3  
Under the same assumptions (`T = "abc"` and all three letters present) any
permutation of `S` that is lexicographically smaller than `A` must contain
`"abc"` as a subsequence.

**Proof.**  
All strings that are lexicographically smaller than `A` share the same
prefix of `a`’s (otherwise they would start with a larger character).  
Consider the first position after the block of `a`’s.

*If a `b` is placed there* (i.e. before the first `c`), then the string
contains `a…b…c` because at least one `c` exists later – exactly the forbidden
subsequence.

*If a character `< 'c'` (i.e. `b`) is placed later but before any `c`,
the same argument applies.*  

The only way to avoid the subsequence is to put **all** `c`’s before the
first `b`, which is exactly what `A` does. Hence any smaller string would
contain `"abc"`. ∎



##### Lemma 4  
If `T = "abc"` but at least one of the letters `a,b,c` is missing from `S`,
the sorted string of `S` does not contain `"abc"`.

**Proof.**  
A subsequence needs one occurrence of each of the three letters.
If one of them is absent, the subsequence cannot exist. ∎



##### Theorem  
For every test case the algorithm outputs the lexicographically smallest
permutation of `S` that does **not** contain `T` as a subsequence.

**Proof.**  
Two cases.

1. **`T` ≠ `"abc"`** or at least one of `a,b,c` missing.  
   By Lemma&nbsp;1 (or Lemma&nbsp;4) the sorted string is safe.
   Any other permutation is lexicographically not smaller than the sorted
   one, therefore the algorithm’s output (the sorted string) is optimal.

2. **`T = "abc"` and all three letters present.**  
   The algorithm outputs the string `A` described in Lemma&nbsp;2.
   Lemma&nbsp;2 guarantees that `A` is safe.
   Lemma&nbsp;3 shows that no safe string can be lexicographically smaller
   than `A`.  
   Hence `A` is the optimal answer.

Both cases cover all possibilities, completing the proof. ∎

**Solution Explanation**

For every test case we are given a circular array  

```
a1 , a2 , … , an   ( a n+1 = a1 )
```

and a string `s` of length `n`.

* `s[i] = 'E'`  →  `ai = a(i+1)`
* `s[i] = 'N'`  →  `ai ≠ a(i+1)`

All numbers may be any positive integers (up to `10^9` is enough).  
We have to decide whether there exists *any* assignment that satisfies all
relations.



--------------------------------------------------------------------

#### 1.   Observations  

* Equality relations are transitive:  
  if `a1 = a2` and `a2 = a3` then also `a1 = a3`.  
  Therefore all positions that are linked by `'E'` must obtain the **same**
  value.  
  We can merge them into one *component*.

* After merging, an inequality `'N'` becomes a condition
  *“the two components must be different”*.  
  This is impossible **only** when the two ends of this `'N'` belong to the
  same component (i.e. we would be forced to say *x ≠ x*).

Consequently the whole problem reduces to:

```
Are there any positions i with s[i] = 'N' whose two ends lie in the same
equality‑component ?
```

If the answer is *no* we can give each component a distinct integer and all
inequalities are satisfied.



--------------------------------------------------------------------

#### 2.   Data structure – Disjoint Set Union (DSU)

The DSU (also called Union‑Find) maintains a partition of the `n` positions
into components and supports

* `union(x, y)` – merge the components of `x` and `y`
* `find(x)` – representative of the component of `x`

Both operations work in almost O(1) time (`α(n)` inverse Ackermann).

Algorithm per test case  

```
make DSU of size n
for i = 0 … n-1
        if s[i] == 'E'          // equality edge
                union(i, (i+1) mod n)

possible = true
for i = 0 … n-1
        if s[i] == 'N'          // inequality edge
                if find(i) == find((i+1) mod n)
                        possible = false
                        break

output "YES" if possible else "NO"
```

The two passes are linear, `n ≤ 50`, so the program is easily fast enough.



--------------------------------------------------------------------

#### 3.   Correctness Proof  

We prove that the algorithm prints **YES** iff a suitable array exists.

---

##### Lemma 1  
After processing all `'E'` characters, two positions belong to the same DSU
set **iff** the constraints force them to be equal.

**Proof.**  
*If* the algorithm united them, there is a chain of `'E'` edges connecting the
two positions, therefore by transitivity they must be equal.  

*Only‑if*: suppose the constraints force `ai = aj`.  
Then there exists a sequence of positions  

```
i = p0 , p1 , … , pk = j
```

such that each consecutive pair `(pr , pr+1)` is linked by an `'E'`
constraint (otherwise equality could not be deduced).  
During the first pass the algorithm unions exactly those pairs, therefore
all `pr` end up in the same DSU set, in particular `i` and `j`. ∎



##### Lemma 2  
If after the first pass there exists an index `i` with `s[i] = 'N'` and
`find(i) = find(i+1)`, then **no** array satisfies all constraints.

**Proof.**  
By Lemma&nbsp;1 `find(i) = find(i+1)` means the constraints already force
`ai = a(i+1)`.  
The character `'N'` additionally requires `ai ≠ a(i+1)`.  
Both cannot hold simultaneously, hence the whole system is inconsistent. ∎



##### Lemma 3  
If for every `'N'` edge the two ends belong to different DSU sets,
then a satisfying array exists.

**Proof.**  
Create one distinct positive integer for each DSU component
(e.g. 1,2,3,…).  
All positions inside a component receive the same integer, therefore every
`'E'` edge is satisfied (by Lemma&nbsp;1).  
For a `'N'` edge the two ends are in different components, consequently they
receive different integers, satisfying the inequality. ∎



##### Theorem  
For a test case the algorithm outputs **YES** exactly when there exists an
array `a` consistent with the given string `s`.

**Proof.**  

*If the algorithm outputs NO.*  
That happens only when it finds an `'N'` edge whose ends are in the same DSU
set. By Lemma&nbsp;2 the constraints are contradictory, so no array exists.

*If the algorithm outputs YES.*  
Then no such contradictory `'N'` edge exists. By Lemma&nbsp;3 we can assign
different numbers to different DSU components and obtain a valid array.
Thus a solution exists.

Both directions hold, establishing the equivalence. ∎

For each test case (`n ≤ 50`)

* first pass: `O(n)` unions,
* second pass: `O(n)` finds.

DSU operations are `α(n)` ≈ constant, therefore  

```
Time   :  O(n)   ≤  O(50) per test case
Memory :  O(n)   (the DSU arrays)
```

Both are negligible for the given limits (`t ≤ 1000`).

```python
import sys

class DSU:
    def __init__(self, n):
        self.parent = list(range(n))
        self.rank = [0] * n

    def find(self, x):
        # path compression
        while self.parent[x] != x:
            self.parent[x] = self.parent[self.parent[x]]
            x = self.parent[x]
        return x

    def union(self, x, y):
        xr = self.find(x)
        yr = self.find(y)
        if xr == yr:
            return
        # union by rank
        if self.rank[xr] < self.rank[yr]:
            self.parent[xr] = yr
        elif self.rank[xr] > self.rank[yr]:
            self.parent[yr] = xr
        else:
            self.parent[yr] = xr
            self.rank[xr] += 1


def solve() -> None:
    data = sys.stdin.read().strip().split()
    if not data:
        return
    t = int(data[0])
    out_lines = []
    idx = 1
    for _ in range(t):
        s = data[idx]
        idx += 1
        n = len(s)
        dsu = DSU(n)

        # unite equality edges
        for i, ch in enumerate(s):
            if ch == 'E':
                dsu.union(i, (i + 1) % n)

        possible = True
        # check inequality edges
        for i, ch in enumerate(s):
            if ch == 'N':
                if dsu.find(i) == dsu.find((i + 1) % n):
                    possible = False
                    break

        out_lines.append("YES" if possible else "NO")

    sys.stdout.write("\n".join(out_lines))
```

For one test case we are given three integer stick lengths  

```

```

Exactly one stick has to be broken into two **positive** integer pieces.
After the break we own four sticks and we must be able to place them as the
four sides of a rectangle (a square is also allowed).

For a rectangle the four side lengths have to consist of two equal lengths
and the other two equal lengths, i.e. after the break the multiset of the
four lengths must be

```
{x , x , y , y}          (x , y > 0,   x may be equal to y)
```

--------------------------------------------------------------------

#### 1.   What can a rectangle look like after the break ?

Let the stick we break be `L`.  
The other two sticks stay unchanged, call them `p` and `q`.

After the break we have the four numbers  

```
p , q , x , L‑x          ( 1 ≤ x ≤ L‑1 )
```

There are only two ways to obtain the pattern `{x,x,y,y}`.

* **Case A – the two whole sticks are already equal**  

  `p == q`.  
  Then we already own one pair of equal sides.
  The two pieces must form the second pair, therefore they have to be equal:

  ```
  x == L‑x   →   L is even and we cut it exactly in half.
  ```

  So a rectangle is possible **iff** two of the original sticks are equal
  **and** the third (the one we break) is even.

* **Case B – the two whole sticks are different**  

  `p != q`.  
  The only way to get two pairs is to let each piece match one of the whole
  sticks:

  ```
  {x , L‑x} = {p , q}
  ```

  Hence `L = p + q`.  
  In words: one stick must have length equal to the sum of the other two.

These two cases are exhaustive – any rectangle must belong to one of them.

--------------------------------------------------------------------

#### 2.   Decision rule

For the three numbers `a , b , c` the answer is **YES** iff at least one of
the following holds

```
a + b == c          (break c into a and b)
a + c == b          (break b into a and c)
b + c == a          (break a into b and c)

a == b and c is even
a == c and b is even
b == c and a is even
```

Otherwise the answer is **NO**.

All checks are O(1), therefore the whole algorithm works in `O(t)` time
(`t ≤ 10^4`) and uses `O(1)` additional memory.

We prove that the algorithm prints “YES” exactly for the feasible test

---

If two of the original sticks are equal and the third stick is even,

**Proof.**  
Assume `a == b` and `c` is even. Break stick `c` into two halves `c/2` and
`c/2`. The four sticks are `a , a , c/2 , c/2` – two pairs of equal lengths,

##### Lemma 2  
If one stick length equals the sum of the other two, then a rectangle can be
built.

**Proof.**  
Assume `c = a + b`. Break stick `c` into pieces `a` and `b`. The four sticks
are `a , b , a , b` – again two equal pairs, forming a rectangle. ∎



##### Lemma 3  
If a rectangle can be built, then either Lemma&nbsp;1 or Lemma&nbsp;2
conditions hold.

**Proof.**  
Let the stick we break be `L` and the untouched sticks be `p` and `q`.  
After the break we have `{p , q , x , L‑x}` and they must be `{x,x,y,y}`.

*If `p == q`*  
the two whole sticks already give one equal pair, therefore the two pieces
must be equal: `x = L‑x`. Consequently `L` is even and we are in the
situation of Lemma&nbsp;1 (the equal whole sticks are `p` and `q`).

*If `p != q`*  
the only way to obtain two pairs is `{x , L‑x} = {p , q}`,
hence `L = p + q`. This is exactly the situation of Lemma&nbsp;2
(the stick we break has length equal to the sum of the other two). ∎

The algorithm outputs “YES” **iff** one of the conditions of Lemma&nbsp;1
or Lemma&nbsp;2 is true.

**Proof.**  
The algorithm checks precisely the six statements listed in the decision
rule, which are exactly the conditions of Lemma&nbsp;1 (three equal‑pair
cases) and Lemma&nbsp;2 (three sum cases). ∎



##### Theorem  
For every test case the algorithm prints “YES” if and only if it is possible

**Proof.**

*If a rectangle can be built* – by Lemma&nbsp;3 the situation satisfies

algorithm is true, and by Lemma&nbsp;4 the algorithm prints “YES”.

Both directions hold, establishing the equivalence. ∎

For every test case we are given  

* a rectangle `n × m` (`1 ≤ n , m ≤ 100`);
* the robot starts in cell `(rb , cb)`;
* the only dirty cell is `(rd , cd)`.

The robot moves each second by the vector `(dr , dc)` :

* initially `dr = 1 , dc = 1`;
* before a move, if the next row would be outside the rectangle,
  `dr` is multiplied by `-1` (reflection on the horizontal wall);
* analogously for the columns and `dc`.

During **every** second (including the moment `t = 0` before the first move)
the robot cleans the whole row **and** the whole column of its current
position.  
The task is to output the smallest `t ≥ 0` when the dirty cell is cleaned,
i.e. when the robot is in the same row **or** the same column as the dirty
cell.



--------------------------------------------------------------------

#### 1.   Motion of one coordinate

Consider only the row coordinate (the column coordinate is completely
analogous).

```
rows are numbered 1 … n
```

If we look at the rows as a line and reflect the rectangle at its borders,
the robot moves on an infinite line with constant speed `+1`.
After every `2·(n‑1)` steps the pattern repeats:

```
periodRow = 2·(n-1)                (if n = 1 the robot never moves vertically)
```

For 0‑based indices (`0 … n-1`) let  

```
r0      = rb-1          (starting row)
target  = rd-1          (row we want to reach)
```

After `t` seconds the (non‑reflected) position on the infinite line is `r0 + t`.
The real row inside the rectangle is

```
p = (r0 + t) mod periodRow
row(t) =  p                     if p < n
          periodRow - p         otherwise
```

The robot is in the wanted row **iff** `row(t) = target`.  
From the definition of `row(t)` this happens for two possible residues of
`(r0 + t) mod periodRow` :

```
p = target                     (going downwards)
p = periodRow - target         (going upwards)
```

Therefore the set of times when the robot is in the wanted row is the union
of two arithmetic progressions

```
t ≡ (target - r0)          (mod periodRow)
t ≡ (periodRow - target - r0) (mod periodRow)
```

The smallest non‑negative element of each progression is simply the
corresponding remainder, i.e.

```
a1 = (target - r0)               mod periodRow
a2 = (periodRow - target - r0)   mod periodRow
```

The same reasoning for the column gives two numbers `b1 , b2`
(using `periodCol = 2·(m-1)`).

--------------------------------------------------------------------

#### 2.   Whole answer

The robot cleans the dirty cell as soon as **either** the row **or** the
column matches.  
Consequently the required time is the minimum of the four candidates

```
answer = min( a1 , a2 , b1 , b2 )
```

Special cases  

* `t = 0` already works if `rb = rd` **or** `cb = cd`.
* If `n = 1` the robot never changes its row – we simply ignore the row
  candidates (they do not exist).  
  The same for `m = 1`.

All numbers are tiny (`period ≤ 198`), the whole computation is O(1) per
test case.

--------------------------------------------------------------------

#### 3.   Correctness Proof  

We prove that the algorithm outputs the smallest time `t` when the dirty
cell is cleaned.

---

##### Lemma 1  
For `n > 1` the set of times when the robot is in row `rd` is exactly

```
{ t ≥ 0 | t ≡ a1 (mod periodRow) } ∪ { t ≥ 0 | t ≡ a2 (mod periodRow) }
```

*Proof.*  
From the bounce description the robot’s row after `t` seconds is
`row(t)` defined in section&nbsp;1.
`row(t) = rd` ⇔ the intermediate value `p = (r0 + t) mod periodRow`
equals either `target` (moving downwards) or `periodRow - target`
(moving upwards).  
Both equalities are linear congruences for `t`. Solving them gives the two
remainders `a1 , a2`. ∎



##### Lemma 2  
If `n = 1` the robot is always in row `1`.  
Hence the dirty cell can be cleaned by a row match **iff**
`rb = rd`, and otherwise the row never contributes a cleaning time.

*Proof.*  
With only one row there is no vertical wall to reflect from, the robot’s
row coordinate never changes. ∎



##### Lemma 3  
Analogous statements of Lemma&nbsp;1 and Lemma&nbsp;2 hold for the column
coordinate.

*Proof.* Identical, replacing `n` by `m`, `r` by `c`. ∎



##### Lemma 4  
Let  

```
Trow = min( a1 , a2 )   (if n > 1, otherwise +∞)
Tcol = min( b1 , b2 )   (if m > 1, otherwise +∞)
```

Then `Trow` (resp. `Tcol`) is the smallest non‑negative time when the robot
shares the dirty cell’s row (resp. column).

*Proof.*  
By Lemma&nbsp;1 the row‑matching times form two arithmetic progressions,
their smallest elements are exactly `a1` and `a2`. The minimum of those two
is the earliest row match. The same argument with Lemma&nbsp;3 gives the
column case. ∎



##### Lemma 5  
The answer produced by the algorithm,
`ans = min( a1 , a2 , b1 , b2 )`,
is the smallest time when the robot cleans the dirty cell.

*Proof.*  
The robot cleans the cell at time `t` **iff** `t` is a row‑matching time
or a column‑matching time.
The set of all such times is the union of the four arithmetic progressions
with remainders `a1 , a2 , b1 , b2`.  
The smallest element of a union of sets of non‑negative integers is the
minimum of the smallest elements of the individual sets, i.e. `min(a1,a2,b1,b2)`.
∎



##### Theorem  
For every test case the algorithm outputs the minimum number of seconds
required for the robot to clean the dirty cell.

*Proof.*  

* If `rb = rd` or `cb = cd` the algorithm returns `0`.

  match occurs.  
  Hence the returned value is the true optimum.
